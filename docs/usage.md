# Usage

End-user playbook for the autonomous spec-queue orchestrator (`/sr-start`). For design details see `docs/specs/ralph-loop-v2-design.md`; for the SKILL contract see `skills/sr-start/SKILL.md`.

## Producing Approved issues

Approved issues are the input to `/sr-start`. The canonical way to produce one is `/sr-spec` — a brainstorming skill that runs the dialogue, lazily creates the issue's per-issue branch+worktree at step 7 (after the design is approved), writes the spec to `docs/specs/<topic>.md` *on that branch* (not on main), runs an adversarial codex review of the spec, overwrites the Linear issue description with the approved spec (preserving the prior description as a comment), sets any `blocked-by` relations identified during design, and transitions the issue to the configured `approved_state`. Invoke `/sr-spec` with an existing issue ID to populate a pre-filed ticket, or without arguments to create the issue at step 6.5. The branch+worktree persist after finalize — `/sr-start` finds them on its next dispatch and reuses them. Other paths to Approved are fine too — hand-editing a description and toggling state works, and the orchestrator's fallback create path will lazily create the branch+worktree on first dispatch — but `/sr-spec` is the path that guarantees the ralph input contract is satisfied and runs the spec-time codex gate.

## When to run `/sr-start`

Run it before stepping away from the desk — typically end-of-day or before a long break — when there are one or more Linear issues in the **Approved** state, across the projects declared in `<repo-root>/.sensible-ralph.json` (either a `projects: [...]` list or an `initiative: "..."` shorthand that expands on every invocation), with a complete PRD (≥200 non-whitespace chars). Workflow fields (state names, labels, worktree base, model) are plugin userConfig values, exported by the Claude Code harness as `CLAUDE_PLUGIN_OPTION_*` env vars; scope is the only field that lives in the per-repo `.sensible-ralph.json`. An issue's blockers are "resolved enough to dispatch" when every blocker is Done, In Review, or **Approved and also in this run's queue** — which means **an entire chain of Approved issues can run end-to-end overnight in a single session**: the orchestrator dispatches the root first; that session completes and moves the root to In Review; then the child dispatches against the parent's branch; and so on down the chain. You don't need to wait for each level to be reviewed before kicking off the next. Blocker chains (`blocks` / `blocked by` relations) are traced across any project in scope — an issue in Project B blocked by an issue in Project A clears automatically when both projects are in `.sensible-ralph.json` — but a blocker whose project is *outside* the scope trips the **out-of-scope blocker** preflight anomaly, which points back at `.sensible-ralph.json` as the fix. The orchestrator dispatches a `claude -p --permission-mode auto` session per issue in DAG-aware order; expect roughly 5–15 minutes of wall time per issue depending on scope. The skill is invoked manually (`disable-model-invocation: true`) and pauses for explicit confirmation after the dispatch preview, so it never auto-runs and never dispatches a queue you haven't seen. Invoke it from a `bash` session anywhere inside the repo — main checkout or a linked worktree both work. Concurrent `/sr-start` sessions in *different* repos are safe by construction (disjoint repo roots → disjoint `.sensible-ralph/progress.json`, worktree dirs, and queue files); concurrent runs in the *same* repo are not supported.

## What to expect in the morning

Inspect `.sensible-ralph/progress.json` at the repo root — it lists every issue the orchestrator touched in this run, keyed by `run_id` (ISO 8601 UTC). Each record has an `event` discriminator (`start` for the dispatch moment, `end` for the final per-issue outcome) and end records carry an `outcome` field. **`in_review`** means the session completed cleanly and `/prepare-for-review` transitioned the issue (review the worktree, then run your project's merge ritual from a fresh session at the main-checkout root — not from inside the worktree). **`failed`** and **`exit_clean_no_review`** mean the issue is labeled with the configured `failed_label` and downstream dependents were tainted with **`skipped`** outcomes — run `/sr-status` to see the inline diagnostic sub-block (the `hint` heuristics, the `transcript:` worktree log path, and the `session:` JSONL transcript path the orchestrator preserved per ENG-308); `cd` into the worktree, read whichever pointer is more useful, then decide whether to retry (remove the failed label and re-queue), cancel the issue, or debug interactively. **`setup_failed`** means the orchestrator never got as far as dispatching `claude -p` (branch lookup, dag_base, worktree creation step, etc., recorded in `failed_step`); cleanup ran for state this invocation created — no diagnostic sub-block here because no session ran. Two outcomes leave Linear untouched and descendants un-tainted because the orchestrator can't tell that anything actually went wrong: **`local_residue`** means the target branch and worktree path are in inconsistent states at the start of dispatch — exactly one exists, or the path is a registered worktree on a different branch (operator state — stale dir from a prior crashed run, in-flight branch with no worktree, or a manually-checked-out worktree on the wrong branch). The orchestrator's stderr names the specific cause (`branch-only`, `path-only`, `wrong-branch`, `path-not-worktree`); the residue is preserved untouched at `residue_path` / `residue_branch`. Under ENG-279's per-issue branch lifecycle this case is rare — the common "both branch and worktree exist" path is reuse, not residue. Clean up by hand and re-queue. **`unknown_post_state`** means claude exited 0 but the post-dispatch Linear state fetch failed transiently; the diagnostic sub-block (hint + transcript paths) renders the same as `failed` so the JSONL is one click away if you want to confirm the session actually reached In Review (success) versus stopped short (re-queue). Triage failed-label and `local_residue` issues before kicking off the next ralph run, otherwise the same blockers will keep tainting (or no-op-skipping) their dependents on every dispatch.

## Checking progress mid-run

Run `/sr-status` from anywhere inside the repo to print a sectioned table — Done / Running / Queued — for the latest ralph run. Read-only, zero side effects, no network calls — just `.sensible-ralph/progress.json` and `.sensible-ralph/ordered_queue.txt`. Useful both during a run (to see what's currently dispatching) and after (to glance at the most recent run's outcomes without having to `jq` `progress.json` by hand). Non-success Done rows render an indented sub-block underneath with the inline `hint`, the worktree log path (`transcript:`), and the JSONL transcript path (`session:`) — operators don't need to remember Claude Code's project-slug naming convention to find the rich session log. The footer prints a `tail` tip pointing at the running issue's stdout log (using the dispatch-time path persisted into the start record, so it stays accurate after `stdout_log_filename` reconfiguration mid-run).

A "Running" row that looks stuck — same elapsed time across multiple `/sr-status` checks — usually means the orchestrator died (Esc in the TUI, terminal closed, machine sleep). Confirm with `pgrep -f orchestrator.sh`. Crash detection lands in ENG-254; until then, verify by hand and clean up via the `local_residue` / `failed` triage paths described above.
